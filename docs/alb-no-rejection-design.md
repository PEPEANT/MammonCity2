# 알바 불합격 제거와 에너지 게이트

현재 결론: 단기알바는 `합격/불합격 연출`보다 `지원 조건 + 에너지 소모/상태`로 읽히게 정리한다. 현재 라이브 코드는 이미 `에너지 > 0 = 100% 합격`, `에너지 = 0 = 실패`에 가깝고, 다음 정리 단계에서는 이 실패 문구를 `에너지 부족으로 지원 불가` 쪽으로 통일한다.

## 목적

- 코드 수정 전에 단기알바 지원 규칙을 문서 기준으로 먼저 고정한다.
- `직장 면접`과 `단기알바 지원`을 서로 다른 감각으로 분리한다.
- 배달 장비 보유를 `확률 보너스`가 아니라 `해금 게이트`로 정리한다.

## 적용 범위

- 이 문서는 `단기알바` 전체 규칙을 다룬다.
- `맥도날드`는 첫 기준 사례지만 예외 시스템으로 두지 않는다.
- `직장` 면접, 정규직 합격률, `career` 상태에는 적용하지 않는다.

## 핵심 규칙

1. 단기알바는 먼저 `장비/자격/장소` 같은 하드 게이트를 확인한다.
2. 하드 게이트를 통과했고 `에너지 > 0`이면 별도 채용 RNG 없이 바로 다음 턴 예약근무를 만든다.
3. `에너지 = 0`이면 예약근무를 만들지 않는다.
4. 이 경우는 `불합격`보다 `지원 불가`로 읽히게 문구를 바꾼다.
5. 하루 1회 지원 제한과 예약근무 1개 제한은 그대로 유지한다.

## 다음 구현 기준

- 단기알바 지원은 장기적으로 `소량 에너지 소모`가 붙는 행동으로 정리한다.
- 즉 `채용 RNG` 대신 `지금 이 컨디션으로 일을 잡을 수 있는가`를 먼저 보여준다.
- 에너지가 바닥난 상태에서는 `불합격` 연출을 만들지 않고, `지원할 기운이 없다`는 needs 메시지를 우선한다.

## UI 문구 기준

- `면접 합격`, `채용 불합격` 같은 표현은 단기알바에서 점차 걷어낸다.
- 단기알바는 아래 상태 문구를 우선한다.
  - `예약됨`
  - `지원 불가`
  - `장비 부족`
  - `에너지 부족`
  - `이미 예약된 근무 있음`
- `직장`만 `지원 -> 면접 -> 합격/불합격` 문법을 계속 쓴다.

## 오토바이 보유와 배달 규칙

- `delivery`는 `bicycle` 또는 `used-motorbike`가 있어야 지원 가능하다.
- `delivery-motorbike`는 `used-motorbike`가 있어야만 지원 가능하다.
- 오토바이 보유는 `합격률 보너스`가 아니라 `오토바이 배달 지원 보장 게이트`로 본다.
- 즉 `used-motorbike`만 있으면 별도 채용 RNG 없이 바로 예약근무가 잡히는 방향이 기준이다.

## 현재 구현 메모

- `getInterviewChanceForOffer()`는 현재 `에너지 > 0 ? 1.0 : 0.0`만 사용한다.
- 즉 확률상으로는 이미 `에너지 > 0`이면 100% 성공이다.
- 다만 `jobs-action-service`와 일부 헤드라인은 아직 `면접 결과`, `면접 합격`, `채용 보류` 같은 구버전 문구를 유지한다.
- 문서 기준은 `채용 RNG 제거`가 먼저고, 남은 작업은 상태 이름과 문구를 이 기준에 맞춰 다시 다듬는 것이다.

## 상태 기준

- `state.jobs.applicationDoneToday`
  하루 1회 단기알바 지원 제한.
- `state.jobs.scheduledShift`
  예약된 다음 턴 근무.
- `state.jobs.interviewResult`
  현재는 호환용 결과 상태로 남아 있지만, 의미상 `단기알바 지원 결과`에 가깝다.
- `used-motorbike`
  오토바이 배달 해금 게이트.

## 리스크와 후속 작업

- 현재 코드/문구에 `면접` 용어가 남아 있어 플레이 감각이 문서와 어긋날 수 있다.
- 단기알바를 무조건 예약형으로 만들면 `에너지` 의미가 약해질 수 있으니, 지원 시점 또는 출근 시점 에너지 소모 연결을 같이 봐야 한다.
- 맥도날드처럼 근무지 도착형 루프는 `지원 성공`보다 `현장 도착`, `대기`, `출근 시작` 쪽 피드백이 더 중요하다.
